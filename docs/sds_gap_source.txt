1. Introduction 
1.1 Purpose 
The purpose of this Software Design Specification (SDS) document is to translate the business requirements 
and functional specifications documented in the University Student Hub (USH) Software Requirements 
Specification (SRS) into a comprehensive technical design that will guide the development team during 
implementation. 
This document serves as: 
 A blueprint for the development team to understand the system architecture, component interactions, and 
data flows 
A reference for quality assurance teams to develop test cases aligned with the design 
A communication tool between stakeholders, ensuring technical decisions are transparent and justified 
Documentation for future maintainers to understand design rationale and system structure 
The intended audience includes: 
Software Developers: To implement the system according to architectural patterns and design decisions 
Database Administrators: To create and optimize the database schema 
UI/UX Designers: To implement interface designs consistent with system architecture 
Quality Assurance Team: To verify that implementation adheres to design specifications 
Project Advisor: To review technical approach and validate design decisions 
Future Maintainers: To understand system architecture for enhancements and troubleshooting 
1.2 General Overview 
System Context 
The University Student Hub (USH) is a web-based platform designed to centralize academic communication, 
resource sharing, and student management for Addis Ababa Institute of Technology (AAiT). The system 
operates as a standalone application that complements existing institutional infrastructure without replacing 
core academic systems such as the Student Information System (SIS) or grade management platforms. 
High-Level Architecture 
USH adopts a Three-Tier Architecture pattern with clear separation of concerns: 

1. Presentation Layer (Client Tier): A React-based Single Page Application (SPA) providing responsive, 
interactive user interfaces accessible via web browsers on desktop and mobile devices. 
2. Application Layer (Business Logic Tier): A Node.js/Express.js RESTful API server handling business 
logic, authentication, authorization, and orchestrating data operations. 
3. Data Layer (Data Tier): PostgreSQL relational database managing persistent storage with proper 
indexing, transactions, and data integrity constraints.

Design Approach 
The design follows industry-standard best practices: 
RESTful API Design: Stateless, resource-oriented endpoints with standard HTTP methods 
JWT-Based Authentication: Secure, scalable token-based authentication eliminating server-side session 
storage 
Role-Based Access Control (RBAC): Granular permission management for different user types  
Repository Pattern: Abstraction layer between business logic and data access 
Service Layer Architecture: Business logic encapsulated in reusable service classes 
Responsive Design: Mobile-first UI approach using Tailwind CSS for consistent styling 
Progressive Enhancement: Core functionality accessible with graceful degradation for older browsers 
Design Goals 
1. Scalability: Support 1,000+ concurrent users with horizontal scaling capability 


2. Maintainability: Modular codebase with clear separation of concerns and comprehensive documentation 
3. Security: Multi-layered security approach protecting user data and preventing common vulnerabilities 
4. Performance: Sub-2-second response times with optimized database queries and efficient caching 
5. Accessibility: WCAG 2.1 Level AA compliance ensuring usability for all students 
6. Reliability: 99.5% uptime during academic hours with automated monitoring and recovery

1.3 Development Methods & Contingencies 
Development Methodology 
The USH project follows an Agile-Scrum hybrid approach adapted for academic constraints: 
Iterative Development: The system will be developed in 2-week sprints with deliverable increments after each 
sprint, allowing for continuous feedback and adjustment. 
Component-Based Architecture: React components and backend services are designed as independent, 
reusable modules that can be developed and tested in parallel by different team members. 
API-First Design: The RESTful API is designed and documented before implementation, allowing frontend 
and backend teams to work concurrently against a defined contract. 
Test-Driven Development (TDD): Critical business logic components (authentication, authorization, data 
validation) are developed using TDD principles with 70%+ code coverage target. 
Design Patterns Applied 
1. Model-View-Controller (MVC): Clear separation between data models (PostgreSQL schemas), views 
(React components), and controllers (Express route handlers) 
2. Repository Pattern: Data access logic abstracted into repository classes, making database operations 
testable and maintainable 
3. Service Layer Pattern: Business logic encapsulated in service classes, called by controllers and reusable 
across multiple endpoints 
4. Factory Pattern: Used for creating complex objects like notification instances with different delivery 
mechanisms 
5. Strategy Pattern: Applied for recommendation algorithms, allowing different recommendation strategies 
to be swapped dynamically 
6. Observer Pattern: Implemented via Socket.io for real-time notification delivery to subscribed clients   
Contingency Plans 

Contingency 1: OpenAI API Rate Limiting or Downtime 
Risk: The AI chatbot depends on external OpenAI API which has rate limits and potential service disruptions. 
Mitigation: 
Implement request queuing and exponential backoff retry logic 
Cache frequently asked questions and responses locally 
Display graceful error messages directing users to alternative support channels 
Maintain fallback FAQ database for common queries 
Contingency 2: Third-Party Authentication Unavailability 
Risk: Google OAuth service disruption could prevent user authentication. 
Mitigation: 
Implement email/password authentication as backup mechanism 
Store encrypted password hashes for users who register with email 
Allow users to link both Google and email credentials 
Provide clear user communication during OAuth service issues 
Contingency 3: Database Performance Degradation 
Risk: As data volume grows, query performance may degrade affecting user experience. 
Mitigation: 
Implement database indexing strategy on frequently queried columns 
Use database query monitoring tools to identify slow queries 
Implement Redis caching layer for frequently accessed data 
Design database schema with partitioning capability for future horizontal scaling 
Contingency 4: Storage Capacity Limitations 
Risk: User-uploaded resources may exceed allocated S3 storage budget. 
Mitigation: 

Implement storage quotas per user and per department 
Provide administrators with storage usage analytics dashboard 
Implement file compression for document types (PDFs, images) 
Archive old resources to cheaper long-term storage tiers 
Implement duplicate file detection to prevent redundant storage 
Contingency 5: Real-Time Notification Scalability 
Risk: WebSocket connections for real-time notifications may not scale to thousands of concurrent users. 
Mitigation: 
Implement connection pooling and load balancing for Socket.io 
Use Redis pub/sub for distributing messages across multiple server instances 
Implement graceful degradation to polling mechanism if WebSocket fails 
Provide email notifications as reliable backup delivery method 
Contingency 6: Development Team Changes 
Risk: Team member unavailability due to academic commitments or other projects. 
Mitigation: 
Comprehensive code documentation with inline comments and README files 
Regular code reviews to ensure knowledge sharing across team members 
Modular architecture allowing components to be worked on independently 
Git branching strategy protecting main codebase from incomplete work 
Weekly sprint meetings to identify blockers and reassign tasks 
Contingency 7: Technology Stack Incompatibility 
Risk: Library version conflicts or deprecated features during development. 
Mitigation: 

Lock dependency versions in package.json to prevent breaking changes 
Regular dependency updates in controlled manner with testing 
Maintain compatibility layer for major version upgrades

Component-Level Decomposition 
Each subsystem is further decomposed into three architectural layers following the MVC pattern: 
Controller Layer: Handles HTTP requests, validates input, calls service layer, and formats responses. 
Service Layer: Contains business logic, orchestrates repository calls, and manages transactions. 
Repository Layer: Abstracts database operations, executes SQL queries, and maps results to domain objects. 
Inter-Subsystem Communication 
Subsystems communicate through well-defined interfaces: 
1. Synchronous Communication: RESTful API endpoints for request-response interactions 
2. Asynchronous Communication: Event-driven messaging via internal event bus for cross-cutting 
concerns 
3. Database Sharing: Controlled access to shared database tables through repository interfaces 

Example: When a new announcement is published: 
1. Announcement Management creates the record 
2. Triggers event: announcement.created 
3. Notification Subsystem listens to event 
4. Sends email/push notifications to target users 
Dependency Management 
To prevent circular dependencies and maintain clean architecture: 
➢ Lower-level subsystems (Auth, Notification) have no dependencies on higher-level subsystems 
➢ Higher-level subsystems (Announcement, Resource) depend on lower-level services 
➢ Shared utilities (validation, logging, error handling) are extracted into separate modules 
➢ Database access flows through repository layer only, never direct queries in controllers

Core Domain Entities 
The system's core domain model includes the following primary entities: 
User: Represents system users (students, instructors, administrators) 
➢ Attributes: userId, email, passwordHash, fullName, role, etc. 
➢ Operations: register(), login(), updateProfile(), changePassword() 
Classroom: Represents academic classrooms/courses 
➢ Attributes: classroomId, name, courseCode, ownerId, invitationCode 
➢ Operations: create(), update(), addMember(), generateInviteCode() 
Announcement: Represents classroom announcements 
➢ Attributes: announcementId, title, content, category, priority 
➢ Operations: create(), publish(), update(), pin() 
Resource: Represents academic resources (notes, assignments, etc.) 
➢ Attributes: resourceId, title, type, fileUrl, avgRating 
➢ Operations: upload(), download(), rate(), delete() 
Schedule: Represents class schedules 
➢ Attributes: scheduleId, courseName, dayOfWeek, startTime, location 
➢ Operations: create(), update(), checkConflict() 
Message: Represents chat messages 
➢ Attributes: messageId, content, status, sentAt 
➢ Operations: send(), markDelivered(), markRead() 
ChatbotConversation: Represents AI chatbot sessions 
➢ Attributes: conversationId, userId, classroomId 
➢ Operations: start(), addMessage() 
Notification: Represents system notifications 
➢ Attributes: notificationId, type, content, isRead 
➢ Operations: create(), markRead() 
 
Key Relationships 
User Relationships: 
➢ User owns 0..* Classrooms 
➢ User is member of 0..* Classrooms (through ClassroomMember) 
➢ User authors 0..* Announcements 
➢ User uploads 0..* Resources 
➢ User manages 0..* Schedules 
➢ User sends 0..* Messages 
➢ User initiates 0..* ChatbotConversations 
➢ User receives 0..* Notifications 
Classroom Relationships: 
➢ Classroom contains 0..* ClassroomMembers 
➢ Classroom has 0..* Announcements

➢ Announcement includes 0..* Attachments

Entity Relationship Summary 
User-Centric Relationships: 
• A User has one UserRole (Student, Instructor, or Administrator) 
• A User participates in multiple Classrooms through ClassroomMembership 
• A User creates multiple Announcements, Resources, Schedules, and Messages 
• A User receives multiple Notifications and Recommendations 
Classroom-Centric Relationships: 
• A Classroom is owned by one User (Instructor) 
• A Classroom contains multiple Announcements, Resources, and Messages 
• A Classroom has multiple Members through ClassroomMembership 
Content Relationships: 
• An Announcement belongs to one Classroom, has one Category, Priority, and Status 
• An Announcement may have multiple Attachments 
• A Resource belongs to one Classroom, has one ResourceType, and multiple ResourceRatings 
• A Schedule belongs to one User and may trigger multiple Reminders 
• A Message belongs to one Classroom and may have replies (parent-child relationship) 
AI Service Relationships: 
• A ChatbotSession belongs to one User and Classroom 
• A ChatbotSession contains multiple ChatbotInteractions 
• A Recommendation links a User to a Resource with confidence scoring

3.3 State Chart Diagrams 
State chart diagrams model the dynamic behavior of individual objects as they transition between states in 
response to events. 
3.3.1 Announcement State Chart 
 
Figure 11: Announcement State Chart Diagram 
State Descriptions: 
Draft State: 
• Entry condition: Instructor creates new announcement or unpublishes existing one 
• Characteristics: Not visible to students, editable without restrictions, no notifications triggered 
• Possible transitions: Publish (to Published), Delete (to termination) 


• Operations allowed: edit(), addAttachment(), removeAttachment(), delete() 
Published State: 
• Entry condition: Instructor publishes draft announcement 
• Characteristics: Visible to all classroom members, triggers notifications based on priority, view counts 
tracked, can be pinned 
• Possible transitions: Expire (automatic when expiration date reached), Archive (manual by admin), 
Unpublish (back to Draft for edits) 
• Operations allowed: edit() (limited), pin(), unpin(), archive() 
• Side effects on entry: Send notifications to classroom members (push + email for URGENT), increment 
announcement count in classroom 
Expired State: 
• Entry condition: System reaches announcement's expiration date 
• Characteristics: Announcement marked as expired, removed from active feed, still searchable in 
archive view 
• Possible transitions: Archive (automatic after 30 days or manual by admin) 
• Operations allowed: archive() 
• Side effects on entry: Remove from home feed, send to expired announcements list 
Archived State: 
• Entry condition: Administrator manually archives or automatic after 30 days in Expired state 
• Characteristics: Hidden from normal views, accessible only via archive search, read-only 
• Possible transitions: Permanent deletion (admin only) 
• Operations allowed: view(), permanentDelete() (admin only) 
3.3.2 User Session State Chart 

 
Figure 12: User Session State Chart Diagram 
State Descriptions: 
Anonymous State: 
• Entry condition: User accesses website without authentication or after logout/session expiration 
• Characteristics: No JWT token, access limited to public landing pages, login/register forms visible 
• Possible transitions: To Authenticating (when login/register initiated) 
• Operations allowed: viewPublicPages(), initiateLogin(), initiateRegistration() 
Authenticating State: 
• Entry condition: User submits login credentials or registration form 
• Characteristics: Temporary state during authentication process, credentials being verified 
• Possible transitions: To Authenticated (success), back to Anonymous (failure) 
• Operations: verifyCredentials(), generateJWT() 
Authenticated State: 


• Entry condition: Successful authentication with valid credentials 
• Characteristics: JWT token issued and stored, user gains access to system features based on role, 
session tracked in Redis cache 
• Possible transitions: To Active (first interaction), to SessionExpired (token expiration), to LoggingOut 
(manual logout) 
• Side effects on entry: Create session record, initialize WebSocket connection, load user preferences 
Active State: 
• Entry condition: Authenticated user interacts with system (< 5 minutes since last action) 
• Characteristics: Full system access, WebSocket connection established for real-time notifications, 
activity timestamps updated 
• Possible transitions: To Idle (inactivity > 5 minutes), to LoggingOut (manual logout) 
• Operations allowed: All operations permitted by user role 
Idle State: 
• Entry condition: No user activity detected for 5 minutes 
• Characteristics: Session still valid, WebSocket connection may be closed to save resources, user can 
resume without re-authentication 
• Possible transitions: To Active (user resumes), to SessionExpired (24 hours of inactivity) 
• Side effects on entry: Close WebSocket connection (if applicable), pause background operations 
SessionExpired State: 
• Entry condition: JWT token expires (24 hours of inactivity) or manual session termination 
• Characteristics: Token invalidated, session data cleared from Redis 
• Possible transitions: To Anonymous (cleanup complete) 
• Side effects on entry: Clear session from Redis, close all connections, invalidate JWT 
• Operations: redirectToLogin() 
LoggingOut State: 
• Entry condition: User manually clicks logout button 
• Characteristics: Temporary state during logout process 
• Possible transitions: To Anonymous (logout complete) 
• Operations: invalidateToken(), clearSession(), closeConnections()

Key Database Design Decisions 
1. UUID Primary Keys: All tables use UUID (type: uuid in PostgreSQL) for primary keys instead of auto-
incrementing integers. This provides: 
• Global uniqueness across distributed systems 
• No sequence collision in multi-instance deployments 
• Security (non-guessable IDs) 
• Future-proofing for potential microservices architecture 
2. Soft Deletes: Critical tables (users, announcements, resources) implement soft deletion patterns: 
• Add deleted_at timestamp column (NULL when active) 
• Queries filter by WHERE deleted_at IS NULL 
• Enables data recovery and audit trails 
• Preserves referential integrity 
3. Timestamps: All tables include: 
• created_at (timestamp with timezone, NOT NULL, DEFAULT NOW()) 
• updated_at (timestamp with timezone, auto-updated via trigger) 
• Enables audit trails and analytics 
4. Indexing Strategy: 
• Foreign keys: Always indexed for JOIN performance 
• Composite indexes: For common query patterns (e.g., classroom_id + status on announcements) 
• Partial indexes: For filtered queries (e.g., WHERE is_active = true) 
• Full-text search: GIN indexes on text columns for efficient search (announcements.content, 
resources.title) 
5. JSON Columns: JSONB type used for: 
• chatbot_interactions.source_citations: Array of citation objects with flexible schema 

• notifications.metadata: Additional context-specific data 
• Provides flexibility without schema migrations 
• Supports indexing and querying with GIN indexes 
6. Enum Types: PostgreSQL native ENUMs for fixed value sets: 
• user_role_enum: STUDENT, INSTRUCTOR, ADMINISTRATOR 
• announcement_category_enum: ACADEMIC, ADMINISTRATIVE, EMERGENCY, EVENT 
• priority_level_enum: NORMAL, IMPORTANT, URGENT 
• announcement_status_enum: DRAFT, PUBLISHED, EXPIRED, ARCHIVED 
• Enforces data integrity at database level 
• Self-documenting schema 
7. Referential Integrity: Foreign key constraints with appropriate ON DELETE behaviors: 
• ON DELETE CASCADE: For dependent records (attachments when announcement deleted) 
• ON DELETE SET NULL: For optional references (parent_message_id in messages) 
• ON DELETE RESTRICT: For critical references (classroom owner_id) 
8. Performance Optimizations: 
• Partitioning strategy for large tables (messages, notifications) by created_at month 
• Materialized views for complex analytics queries 
• Connection pooling (pgBouncer) with max 100 connections 
• Query result caching in Redis for frequently accessed data 
Sample SQL Schema (Users Table) 
CREATE EXTENSION IF NOT EXISTS "uuid-ossp"; 
 
CREATE TYPE user_role_enum AS ENUM ('STUDENT', 'INSTRUCTOR', 'ADMINISTRATOR'); 
 
CREATE TABLE users ( 
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(), 
    email VARCHAR(255) NOT NULL UNIQUE, 
    password_hash VARCHAR(255) NOT NULL, 
    full_name VARCHAR(100) NOT NULL, 
    department VARCHAR(50) NOT NULL, 
    year_of_study INTEGER CHECK (year_of_study BETWEEN 1 AND 5), 
    phone_number VARCHAR(20), 
    profile_picture TEXT, 
    role user_role_enum NOT NULL DEFAULT 'STUDENT', 
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(), 
    last_login TIMESTAMP WITH TIME ZONE, 
    is_verified BOOLEAN NOT NULL DEFAULT FALSE, 
    deleted_at TIMESTAMP WITH TIME ZONE, 
     
    CONSTRAINT email_format CHECK (email ~* '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-
z]{2,}$') 
); 
 
CREATE INDEX idx_users_email ON users(email) WHERE deleted_at IS NULL; 
CREATE INDEX idx_users_role ON users(role) WHERE deleted_at IS NULL; 
CREATE INDEX idx_users_department ON users(department) WHERE deleted_at IS NULL; 
 
-- Auto-update updated_at timestamp 

CREATE OR REPLACE FUNCTION update_updated_at_column() 
RETURNS TRIGGER AS $$ 
BEGIN 
    NEW.updated_at = NOW(); 
    RETURN NEW; 
END; 
$$ language 'plpgsql';
